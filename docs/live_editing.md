## 1. The problem File-Based Live Editing solves

A raw prompt like

> "buat RC low-pass 1 kHz dengan C 100 nF, simulasikan, jelaskan
> perhitungannya"

asks for several different kinds of work: requirement extraction, a
closed-form calculation, value picking, schematic capture, simulation,
result interpretation, and explanation. If the AI agent is allowed to
write `.asc` coordinates or `.cir` directives by hand, every one of
those steps is a place where a silent bug can land.

File-Based Live Editing constrains the agent in three ways:

1. **The Circuit Graph is the source of truth.** Every change the
   agent makes goes through a validated `CircuitGraph` document
   (`circuit.graph.json`). The agent never edits `.asc` text or
   `.cir` text directly.
2. **Math, not vibes, picks the numbers.** The math core is the only
   place that turns a target like "1 kHz cutoff" into a concrete
   `R = 1.6 kΩ`. The agent names the topology and the targets; the
   math core returns a deterministic `FormulaResult` and an
   `E24` lookup. See [`math_core.md`](math_core.md).
3. **Every state change is auditable.** Each edit operation is logged
   to `edit_history.jsonl`, every checkpoint produces a numbered
   snapshot under `.snapshots/`, and the final design must clear a
   structured verification gate (`verification.json`) before it is
   declared done.

The high-level flow is in plan §4.2:

```text
prompt
  -> requirement.json
  -> circuit.graph.json       (Circuit Graph, primary editing model)
  -> circuit.ir.json          (validated generation contract)
  -> circuit.cir              (SPICE netlist, simulation input)
  -> circuit.asc              (schematic, human view)
  -> circuit.plt              (plot settings)
  -> LTspice run
  -> circuit.log / circuit.raw
  -> result.json
  -> calculation.md / calculation.json
  -> verification.json
```

The agent's job is the right-hand side of the arrows. The Python
core's job is the left-hand side.

## 2. Source-of-truth ordering

From plan §4.3, lowest to highest authority:

1. `circuit.graph.json` — primary editing model. The agent reads and
   writes this.
2. `circuit.ir.json` — the stable `CircuitIR` contract from Phase 1.
   The graph lowers into the IR; the rest of the pipeline consumes
   the IR.
3. `circuit.cir` — SPICE netlist. Generated from the IR by
   `ltagent.netlist.render_netlist`.
4. `circuit.asc` — human-viewable schematic. Generated by
   `ltagent.asc.render_asc` with deterministic per-topology placers.
5. `result.json` — measurement outcomes from `ltagent.log_parser`.
6. `calculation.json` / `calculation.md` — math report produced by
   `ltagent.math_core.calculation_report`.
7. `verification.json` — pass/fail summary produced by
   `ltagent.live.verification`.

`.asc` is **not** a source of truth because it embeds coordinate and
routing data that is fragile to edit by hand. The agent must not
attempt to write `.asc` lines directly; the `.asc` writer is
deterministic, and that property is a project invariant (see
`AGENTS.md` §"Hard rules for agents" item 1).

## 3. Project directory layout

Every live project lives under the configured `projects_root` as a
slug-safe directory named after the project id. The layout matches
plan §6:

```text
projects/<project_id>/
  circuit.graph.json      primary editing model (Circuit Graph)
  circuit.ir.json         Circuit IR v0.1
  circuit.cir             SPICE netlist
  circuit.asc             LTspice schematic
  circuit.plt             plot settings (planned)
  metadata.json           human-readable summary
  calculation.md          math derivation (planned for live runs)
  calculation.json        machine-readable math report (planned)
  result.json             last simulation result
  verification.json       verification gate outcome after a verified run
  edit_history.jsonl      append-only audit log
  preview.svg             optional schematic preview (planned)
  preview.png             optional raster preview (planned)
  circuit.log             raw simulation log (after a run)
  circuit.raw             raw waveform data (after a run, gitignored)
  .snapshots/
    001_initial/
    002_before_set_r1/
    ...
```

The directory is created by `ltagent.live.project.create_live_project`
(plan §6, Agent 3's surface). The file-name constants live in
`src/ltagent/live/project.py` as module-level `Final[str]`; any new
artefact added to the layout needs a matching constant there.

### 3.1 `metadata.json`

```json
{
  "projectId": "rc_lowpass_1khz",
  "name": "RC Low-pass 1 kHz",
  "createdAt": "2026-06-19T12:00:00+07:00",
  "updatedAt": "2026-06-19T12:10:00+07:00",
  "createdBy": "ltagent",
  "supportLevel": "official_template",
  "mode": "reliable",
  "topology": "rc_lowpass",
  "status": "verified",
  "files": {
    "graph": "circuit.graph.json",
    "ir":    "circuit.ir.json",
    "netlist": "circuit.cir",
    "schematic": "circuit.asc",
    "plot": "circuit.plt",
    "calculation": "calculation.md",
    "result": "result.json"
  }
}
```

### 3.2 `edit_history.jsonl`

Append-only. One JSON object per line:

```jsonl
{"step":1,"time":"2026-06-19T12:00:00+07:00","op":"create_project","reason":"initial prompt","prompt":"buat RC low-pass 1kHz"}
{"step":2,"time":"2026-06-19T12:02:00+07:00","op":"set_component_value","target":"R1","old":"1.59k","new":"1.6k","reason":"select E24 standard value"}
{"step":3,"time":"2026-06-19T12:03:00+07:00","op":"run_simulation","success":true,"measurements":{"fc":"994.7Hz"}}
```

The history file is never truncated or rewritten. A `git bisect`-style
reviewer can replay the entire edit history by reading the file
sequentially.

## 4. Snapshot and undo

Every state-changing operation calls `snapshot.create` *before*
applying the edit. The snapshot is a copy of the current set of live
files plus a manifest:

```text
.snapshots/
  001_initial/
    circuit.graph.json
    circuit.ir.json
    circuit.cir
    circuit.asc
    metadata.json
    calculation.json
    result.json
    manifest.json
```

The manifest describes why the snapshot exists:

```json
{
  "snapshotId": "001_initial",
  "createdAt": "2026-06-19T12:00:00+07:00",
  "reason": "before edit: set_component_value R1",
  "files": [
    "circuit.graph.json",
    "circuit.ir.json",
    "circuit.cir",
    "circuit.asc",
    "metadata.json"
  ]
}
```

### 4.1 Commands (CLI surface — plan §10)

These will be wired by Agent 3 / Agent 0 once the orchestration
layer lands. **[Planned]** in the sense that the canonical command
names are defined; the executable surface is implemented as the live
orchestration lands.

```bash
ltagent live snapshot  projects/demo --reason "before add opamp" --json
ltagent live snapshots projects/demo --json
ltagent live restore   projects/demo 002_before_add_opamp --json
ltagent live diff      projects/demo 001_initial 003_final --json
```

All commands return the standard envelope (`success`, `command`,
`message`, `data`, `warnings`, `errors`).

### 4.2 Restore semantics

`live restore <project_id> <snapshot_id>` rehydrates the live files
from the snapshot and **does not delete the snapshot**. Snapshots are
immutable: once written, the bytes in `.snapshots/<id>/` never change.
A restore copies them back over the live files; the snapshot
directory remains intact so the user can roll forward again.

### 4.3 Diff semantics

`live diff <project_id> <from_id> <to_id>` walks the two snapshot
directories and returns a structured per-file diff (added, removed,
modified, byte-equal). The diff is **read-only** and never touches the
live tree. The diff payload is the input the agent uses when it has
to explain *why* the design changed between two checkpoints.

## 5. Edit operations

The agent never edits a file directly. Every change is expressed as
an edit operation that the live core applies atomically (snapshot →
graph mutation → IR lowering → netlist/schematic regeneration →
history append).

### 5.1 Operation shape

```json
{
  "op": "set_component_value",
  "projectId": "rc_lowpass_1khz",
  "args": {
    "componentId": "R1",
    "value": "1.6k"
  },
  "reason": "Adjust cutoff frequency to 1kHz using E24 standard value"
}
```

### 5.2 MVP operation catalog (plan §8.3)

```text
open_project
inspect_circuit
add_component
remove_component
replace_component
set_component_value
rename_component
connect
disconnect
rename_net
add_ground
add_label
add_directive
add_measurement
add_probe
generate_files
run_simulation
verify_project
```

### 5.3 Example operations

#### Add a component

```json
{
  "op": "add_component",
  "args": {
    "id": "R3",
    "kind": "resistor",
    "value": "10k",
    "pins": {"1": "out", "2": "fb"},
    "role": "feedback_resistor"
  }
}
```

#### Connect a pin to a net

```json
{
  "op": "connect",
  "args": {
    "componentId": "U1",
    "pin": "OUT",
    "net": "vout"
  }
}
```

#### Add a measurement

```json
{
  "op": "add_measurement",
  "args": {
    "name": "VOUT_MAX",
    "analysis": "tran",
    "expression": "MAX V(out)"
  }
}
```

### 5.4 Apply workflow (plan §8.5)

Every operation flows through the same pipeline:

```text
1. Validate edit op schema.
2. Load project.
3. Create snapshot.
4. Apply edit to graph.
5. Validate graph.
6. Convert graph -> IR.
7. Generate .cir.
8. Generate .asc.
9. Lint schematic / layout.
10. Append to edit_history.jsonl.
11. Return structured result.
```

The apply pipeline is the **only** sanctioned path from "user intent"
to "files on disk". Direct file writes that skip the pipeline are a
review failure.

### 5.5 Error shape

When validation fails, the apply pipeline returns the standard
envelope with a structured error:

```json
{
  "success": false,
  "command": "live.apply",
  "message": "Graph validation failed",
  "data": {"projectId": "demo"},
  "warnings": [],
  "errors": [
    {
      "code": "FLOATING_NODE",
      "detail": "Node fb has only one connected pin",
      "data": {"node": "fb"}
    }
  ]
}
```

Error codes are stable strings (see `ltagent.live.graph_validation`
issue types). Agents must switch on the `code`, not the prose.

## 6. Worked workflow

The same example that opens the plan (§0), traced through the live
editing surface.

### 6.1 Initial prompt

```text
buat RC low-pass 1 kHz dengan C 100 nF, simulasikan, dan jelaskan
perhitungannya
```

### 6.2 Step-by-step

```text
1.  open_project rc_lowpass_1khz
    -> returns { success: true, data: { graph: <empty scaffold> } }

2.  add_component Vin   kind=voltage_source  value="SINE(0 1 1k)"  pins {+:"in", -:"0"}
3.  add_component R1    kind=resistor        value="1591.55"      pins {1:"in", 2:"out"}  role=series_resistor
4.  add_component C1    kind=capacitor        value="100n"          pins {1:"out", 2:"0"}  role=shunt_capacitor
5.  add_directive ".ac dec 100 10 100k"
6.  add_measurement GAIN_1K analysis=ac expression="FIND mag(V(out)/V(in)) AT=1k"

7.  math_core: rc_lowpass_resistor(fc=1000, C=100e-9)
    -> { ideal_R: 1591.55, error_percent: 0.528 }
    math_core: nearest_standard_value(1591.55, "E24")
    -> { selected: 1600, error_percent: 0.528 }

8.  set_component_value R1 value="1.6k"  reason="E24 standard value"
9.  generate_files                  (graph -> IR -> .cir -> .asc)
10. run_simulation                  (LTspice batch, returns RunResult)
11. verify_project                  (verification engine vs. targets)
12. write calculation.json + calculation.md
13. append edit_history.jsonl       (every step above is logged)
```

### 6.3 Final state

```text
projects/rc_lowpass_1khz/
  circuit.graph.json   # source of truth
  circuit.ir.json      # CircuitIR v0.1
  circuit.cir          # generated, simulated
  circuit.asc          # generated, layout-checked
  result.json          # {fc_meas: 994.7, ...}
  verification.json    # {overall_passed: true, confidence: 0.94}
  calculation.json     # {ideal_R: 1591.55, selected_R: 1600, ...}
  calculation.md       # human derivation
  edit_history.jsonl   # 13 lines, one per step above
  .snapshots/
    001_initial/       # before any component added
    002_before_set_r1/ # before the E24 snap to 1.6k
    003_before_simulate/  # before the LTspice run
```

The agent can now answer "kenapa cutoff-nya 994.7 Hz dan bukan 1000?"
by reading `calculation.md` and pointing at the 0.528 % ideal-vs-E24
error.

## 7. Editing an existing project

The follow-up case from plan §0:

```text
User: "buka project amplifier_01, cari kenapa output clipping, lalu perbaiki"
```

Flow:

```text
1.  open_project amplifier_01
2.  inspect_circuit                         -> graph, IR, last result
3.  snapshot(reason="before diagnosis")     -> .snapshots/NNN_diagnosis/
4.  identify gain network (Rin, Rf)
5.  math_core: noninv_feedback(Av=10, Rg=10k)
    -> { Rf_ideal: 90k, Rf_selected: 91k, error_percent: +1.1 }
6.  set_component_value Rf value="91k"  reason="reduce gain to avoid clipping"
7.  generate_files + run_simulation
8.  verify_project (target: no clipping, gain still >= 8)
9.  if failed: revert via live restore <snapshot_id>
10. if passed: append history, leave live files in their verified state
```

Restore at step 9 is the rollback path. The snapshot from step 3 is
what makes the rollback safe.

## 8. MCP live-editing tools (plan §11)

The MCP server exposes eight live/math tools. Tool bodies live in
`ltagent.mcp_live_tools` and are registered in `ltagent.mcp_server`.
No new tools may be added that bypass `ltagent.security` path
validation or that write `.raw` files.

### 8.1 Implemented tools

```text
live_open_project
live_inspect_project
live_apply_edit
live_snapshot
live_restore_snapshot
live_run_and_verify
calculate_circuit
explain_calculation
```

`live_apply_edit` regenerates IR/netlist/schematic when the graph is
complete enough for the existing deterministic generators. Otherwise
it returns `LIVE_GENERATION_NOT_RUN` instead of claiming the derived
files are current.

### 8.2 Planned tools

```text
live_diff
live_optimize
design_circuit
edit_circuit
```

### 8.3 Design principle

> MCP tools harus kecil, eksplisit, schema-driven, path-safe, tidak
> memberi arbitrary shell, tidak memberi arbitrary file read/write,
> return structured JSON, snapshot before modification.
> (plan §11.1)

Every tool returns the standard envelope. Mutating operations snapshot
before persistence, remain under the configured projects root, and
return structured validation failures.
